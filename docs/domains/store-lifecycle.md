# Store Lifecycle — Жизненный цикл кабинета в R5

**Version:** 1.0
**Date:** 2026-03-11
**Status:** Active Domain Logic

---

## Назначение документа

Этот документ определяет **этапы работы с клиентом** в системе R5.

Этапы (stages) отражают реальный прогресс работы с кабинетом маркетплейса и используются для:
- Операционного управления (где клиент в воркфлоу)
- Фильтрации и сортировки в UI
- Аналитики (retention funnel, conversion по этапам)
- Автоматизаций (notifications, alerts, workflow triggers)

**Важно:** Этапы управляются **вручную менеджером** на основе реального состояния работы с клиентом.

---

## Этапы работы с клиентом

### 1. 📄 Договор (`contract`)

**Описание:**
Есть карточка компании и подписанный договор, но доступы к маркетплейсу ещё не получены.

**Критерий перехода на этот этап:**
- Договор подписан
- Кабинет создан в системе (stores record exists)
- API-ключи ещё не получены

**Что видим в системе:**
- Запись в таблице `stores`
- Поля `api_token`, `client_id`, `api_key` пустые или невалидные
- Нет синхронизированных данных (products, reviews, chats)

**Действия менеджера:**
- Ожидаем получения доступов от клиента
- Можем настроить контактные данные, создать заметки

**Следующий этап:** Получены доступы (когда клиент передаст API-ключи)

---

### 2. 🔑 Получены доступы (`access_received`)

**Описание:**
Клиент передал API-ключи от маркетплейса. Можем подключиться к кабинету, но первая синхронизация ещё не выполнена.

**Критерий перехода на этот этап:**
- API-ключи получены и сохранены в системе
- Валидация токенов пройдена (можем делать API-запросы)
- Первая синхронизация ещё не запущена или в процессе

**Что видим в системе:**
- `stores.api_token`, `client_id`, `api_key` заполнены
- Синхронизация может быть в процессе (pending)
- Данные ещё не загружены или частично загружены

**Действия менеджера:**
- Запустить первую синхронизацию (products, reviews, chats)
- Проверить корректность подключения

**Следующий этап:** Кабинет подключён (после успешной первой синхронизации)

---

### 3. ✅ Кабинет подключён (`cabinet_connected`)

**Описание:**
Кабинет успешно подключён к R5, первая синхронизация прошла. Видим товары, отзывы, чаты. Готовы начинать работу.

**Критерий перехода на этот этап:**
- Первая синхронизация завершена успешно
- Данные загружены: products, reviews, chats
- Система готова к работе

**Что видим в системе:**
- Есть записи в `products` (товары загружены)
- Есть записи в `reviews` (отзывы загружены)
- Есть записи в `chats` (чаты загружены, если были)
- `stores.total_reviews`, `total_chats` заполнены

**Действия менеджера:**
- Настроить `product_rules` (какие товары в работе)
- Запустить подачу жалоб (WB) или открытие чатов (OZON)

**Следующий этап:** Поданы жалобы (когда начнём работу)

---

### 4. 📋 Поданы жалобы (`complaints_submitted`)

**Описание:**
Начали подавать жалобы на негативные отзывы (WB) или открывать чаты (OZON). Работа запущена.

**Критерий перехода на этот этап:**
- **WB:** Хотя бы одна жалоба подана (запись в `complaint_details`)
- **OZON:** Хотя бы один review-linked чат открыт (запись в `review_chat_links`)

**Что видим в системе:**
- **WB:** Есть записи в `complaint_details` или `review_complaints`
- **OZON:** Есть записи в `review_chat_links`
- Метрика "Обработанные отзывы" > 0

**Действия менеджера:**
- Мониторить прогресс подачи жалоб
- Подготовить чаты для отзывов, которые не скрыты через жалобы

**Следующий этап:** Открыты чаты (когда начнём вести диалоги)

---

### 5. 💬 Открыты чаты (`chats_opened`)

**Описание:**
Активно работаем в чатах с покупателями — ведём диалоги, предлагаем удаление отзывов, решаем проблемы клиентов.

**Критерий перехода на этот этап:**
- Хотя бы один review-linked чат открыт
- Идёт активная переписка (есть сообщения от R5)
- Менеджеры работают с очередью чатов

**Что видим в системе:**
- Есть записи в `review_chat_links`
- Есть сообщения в `chat_messages` с `is_from_client = FALSE` (наши сообщения)
- Метрика "Открытые чаты" > 0

**Действия менеджера:**
- Вести диалоги через TG Mini App или Web Dashboard
- Отслеживать прогресс по deletion workflow (tags: offered, agreed, confirmed)
- Запускать auto-sequences при необходимости

**Следующий этап:** Текущий контроль (когда основная работа завершена, прогресс >80%)

---

### 6. 🟢 Текущий контроль (`monitoring`)

**Описание:**
Клиент на постоянном обслуживании. Жалобы поданы, чаты открыты, система работает в штатном режиме. Мониторим новые отзывы и поддерживаем процесс.

**Критерий перехода на этот этап:**
- Основная работа завершена (прогресс по жалобам/чатам >80%)
- Клиент в активном статусе
- Регулярная синхронизация работает
- Нет критических backlog'ов

**Что видим в системе:**
- Высокий процент обработанных отзывов (>80%)
- Высокий процент открытых чатов (>80%)
- Регулярные синхронизации (products/reviews/chats) без ошибок
- Возможно автоматизированные процессы (auto-sequences, notifications)

**Действия менеджера:**
- Мониторить новые отзывы
- Реагировать на новые негативные отзывы (жалобы + чаты)
- Отслеживать метрики (deletion rate, response time)
- Поддерживать коммуникацию с клиентом

**Следующий этап:**
- Остаёмся в `monitoring` (стабильное состояние)
- Или → `client_paused` (временная пауза по запросу клиента)
- Или → `client_lost` (отток клиента)

---

### 7. ⏸️ Клиент на паузе (`client_paused`)

**Описание:**
Временная приостановка работы по инициативе клиента. Клиент не отказался от сотрудничества, но по каким-то причинам (финансовые, операционные, сезонность) просит временно не вести работу.

**Критерий перехода на этот этап:**
- Клиент попросил приостановить работу
- Договор не расторгнут
- Планируется возобновление работы

**Причины паузы (примеры):**
- Финансовые трудности клиента (не может оплачивать сейчас)
- Сезонность бизнеса (товары не в сезон, нет смысла работать)
- Внутренние проблемы клиента (реорганизация, смена команды)
- Ожидание решения технических проблем (например, WB чаты отключены)

**Что видим в системе:**
- Синхронизации могут продолжаться (для мониторинга)
- Новые жалобы не подаём
- Новые чаты не открываем
- Существующие чаты можем закрывать
- Auto-sequences останавливаются

**Действия менеджера:**
- Не инициировать новую работу (жалобы, чаты)
- Поддерживать контакт с клиентом (когда планирует вернуться)
- Мониторить состояние кабинета (пассивно)

**Следующий этап:**
- → Вернуться на предыдущий этап (когда клиент готов возобновить)
- → `client_lost` (если пауза затянулась и клиент не возвращается)

**Технические действия при паузе:**
- Остановить auto-sequences (`is_active = FALSE`)
- Закрыть активные чаты или оставить без новых действий
- Не подавать новые жалобы
- Синхронизации: опционально (можно продолжать для мониторинга, можно остановить)

---

### 8. 🔴 Клиент потерян (`client_lost`)

**Описание:**
Клиент отключился, прекратил сотрудничество или ушёл к конкурентам. Работа с кабинетом полностью остановлена.

**Критерий перехода на этот этап:**
- Клиент расторг договор
- Клиент перестал отвечать (churn)
- Клиент явно отказался от услуг
- Пауза затянулась >3-6 месяцев без возврата

**Что видим в системе:**
- Все синхронизации остановлены
- Все auto-sequences остановлены
- Чаты закрыты или заморожены
- API-токены могут быть отозваны клиентом

**Действия менеджера:**
- Остановить все автоматизации
- Архивировать данные (опционально)
- Зафиксировать причину оттока (для аналитики)
- Закрыть все активные задачи по кабинету

**Следующий этап:**
- Обычно финальный этап (клиент ушёл)
- Редко: возврат на `contract` (если клиент вернулся через время)

**Технические действия при оттоке:**
- Остановить cron синхронизации для этого store
- Остановить все auto-sequences
- Закрыть все чаты (`status = 'closed'`, `completion_reason = 'client_lost'`)
- Опционально: пометить `stores.status = 'archived'` (если будем использовать status для архивации)

## Управление этапами

### Кто управляет этапами?

**Manual control только:**
- Менеджер переставляет этапы вручную на основе реального состояния работы
- Никакой автоматики (на текущем этапе развития системы)

### Как менять этап?

**UI:**
- Web Dashboard: dropdown selector на карточке store
- TG Mini App: опционально (если нужно)

**API:**
- `PATCH /api/stores/:id/stage`
- Body: `{ stage: 'monitoring' }`
- Permissions: owner, admin, manager (только для своих stores)

### Audit trail:

Все изменения этапа логируются (будущая фича):
- `store_stage_history` table (аналогично `chat_status_history`)
- Поля: `store_id`, `old_stage`, `new_stage`, `changed_by`, `changed_at`, `change_source`

## Интеграция с системой

### Где используются этапы?

#### 1. UI / Dashboard
- Фильтр по этапам (sidebar)
- Сортировка по этапам
- Колонка "Этап" в таблице stores
- Badge с цветовой индикацией

#### 2. Notifications
- Alert, если клиент застрял на этапе >X дней

- Weekly report по distribution этапов

#### 3. Analytics
- Retention funnel: сколько клиентов доходят до `monitoring`
- Conversion rate между этапами

- Churn analysis: на каком этапе теряем клиентов

#### 4. Automation (future)
- Автоматические переходы на основе метрик (опционально)
- Workflow triggers (например, если `stage = 'contract'` >7 дней → notify sales)

## Связь с техническими статусами (Status)

### Текущая ситуация (до миграции):

В системе есть поле `stores.status`:
- `active` — система работает
- `trial` — пробный период
- `paused` — на паузе
- `stopped` — остановлен
- `archived` — архивирован

**Проблема:**
- `status` используется в extensions, cron jobs, dashboard
- Нельзя просто удалить или заменить на `stage`

### План миграции:

**Phase 1 (текущий Sprint-006):**
- Добавляем поле `stores.stage`
- Используем `stage` параллельно с `status`
- `status` остаётся для обратной совместимости

**Phase 2 (Sprint-007 или позже):**
- Audit всех использований `stores.status`
- Постепенная замена `status` на `stage` в коде
- Extensions переписываем: `status = 'active'` → `stage NOT IN ('client_lost', 'client_paused')`
- Cron jobs переписываем: `status IN ('active', 'trial')` → `stage IN (...)`

**Phase 3 (будущее):**
- Deprecate поле `stores.status`
- Полный переход на `stores.stage`

**Детальный план:** см. `docs/sprints/Sprint-006/BACKLOG.md`

## Enum значения (для DB)

### PostgreSQL Type:

```sql
CREATE TYPE store_stage AS ENUM (
  'contract',              -- Договор
  'access_received',       -- Получены доступы
  'cabinet_connected',     -- Кабинет подключён
  'complaints_submitted',  -- Поданы жалобы
  'chats_opened',          -- Открыты чаты
  'monitoring',            -- Текущий контроль
  'client_paused',         -- Клиент на паузе
  'client_lost'            -- Клиент потерян
);
```

Или проще VARCHAR(50) с constraint:

```sql
ALTER TABLE stores
  ADD COLUMN stage VARCHAR(50) DEFAULT 'cabinet_connected',
  ADD CONSTRAINT stores_stage_check
    CHECK (stage IN (
      'contract',
      'access_received',
      'cabinet_connected',
      'complaints_submitted',
      'chats_opened',
      'monitoring',
      'client_paused',
      'client_lost'
    ));
```

## Business Rules

### Правила перехода между этапами:

**Типичный success path:**
```
contract
  → access_received
    → cabinet_connected
      → complaints_submitted
        → chats_opened
          → monitoring (стабильное состояние)
```

**Альтернативные пути:**

**Пауза:**
```
любой этап → client_paused → возврат на предыдущий этап
```

**Отток:**
```
любой этап → client_lost (финальный)
```

**OZON path (без жалоб):**
```
contract
  → access_received
    → cabinet_connected
      → chats_opened (сразу, минуя complaints_submitted)
        → monitoring
```

### Ограничения (optional):

**Запрещённые переходы (validation):**
- `client_lost` → любой другой этап (кроме `contract` при возврате клиента)
- Нельзя перескочить этап (например, `contract` → `monitoring`)

**Рекомендация:** На первом этапе не делать жёсткую валидацию, дать менеджерам свободу.


## FAQ

### Q: Когда переставлять этап вручную?

**A:** Менеджер переставляет этап когда:
- Видит реальное изменение в работе (начали подавать жалобы → `complaints_submitted`)
- Клиент попросил паузу → `client_paused`
- Клиент ушёл → `client_lost`
- Достигли стабильного состояния → `monitoring`

### Q: Можно ли автоматизировать переходы?

**A:** Можно, но не сейчас. На Phase 1 (Sprint-006) всё ручное. Позже можем добавить:
- Auto-transition: `cabinet_connected` → `complaints_submitted` когда `reviews_processed > 0`
- Auto-transition: `chats_opened` → `monitoring` когда прогресс >80%

Но это требует тестирования и валидации.

### Q: Что делать если этап не подходит?

**A:** Использовать ближайший подходящий этап. Система гибкая, этапы — это guidance, а не strict constraint.

### Q: Клиент на паузе vs Клиент потерян?

**A:**
- **Пауза:** Временно, планируем вернуться, договор активен
- **Потерян:** Окончательно, договор расторгнут или клиент пропал
