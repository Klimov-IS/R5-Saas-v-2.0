# Sprint 010 — Аудит: Фильтр по дате в задачах расширения

**Дата:** 2026-03-14

---

## Контекст

В системе есть два уровня фильтрации отзывов по дате:

1. **Глобальный cutoff WB (`2023-10-01`):** Отзывы до этой даты не влияют на рейтинг товара. WB не принимает жалобы на такие отзывы, чаты по ним открыть невозможно. Парсинг статусов бесполезен.

2. **Per-product `work_from_date`:** Каждый товар имеет настройку в `product_rules.work_from_date` (по умолчанию `2023-10-01`). Это дата, с которой клиент решил брать отзывы в работу. Позволяет сузить диапазон — например, начать работу только с отзывов 2025 года.

Оба фильтра объединяются через `COALESCE(pr.work_from_date, '2023-10-01')`.

## Найденная проблема

Фильтр по дате уже применялся при **генерации жалоб** (`helpers.ts`, `generate-complaints-batch`), но **НЕ применялся** в задачах расширения:

| Запрос | Фильтр по дате | Результат |
|--------|:--------------:|-----------|
| Query A (statusParses) | **Нет** | Расширение парсило отзывы 2020-2022 |
| Query E (status_parses_total) | **Нет** | Счётчик завышен |
| Q2 stores (statusParses indicator) | **Нет** | Индикатор "есть задачи" для старых отзывов |
| Генерация жалоб (helpers.ts) | **Да** | Работает корректно |
| DB constraint (review_complaints) | **Да** | CHECK >= 2023-10-01 |

## Влияние

- Расширение тратило время на парсинг страниц для отзывов, по которым ничего сделать нельзя
- Счётчик `statusParses` был завышен — показывал тысячи задач, которые на самом деле не нужны
- Индикатор в `GET /stores` показывал "есть задачи" для магазинов, где все непарсенные отзывы — старые

## Решение

Добавлено условие `AND r.date >= COALESCE(pr.work_from_date, '2023-10-01')` в три запроса:
- Query A (statusParses) — tasks/route.ts
- Query E (status_parses_total) — tasks/route.ts
- Q2 (statusParses indicator) — stores/route.ts

Запросы chatOpens, chatLinks, complaints не затронуты — они защищены транзитивно (зависят от данных, которые уже фильтруются по дате при создании).
