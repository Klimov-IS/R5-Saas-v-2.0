## 1. Общая концепция работы в чатах

### 1.1 Цель

Связаться с покупателем, оставившим негативный отзыв, через чат маркетплейса. Узнать его проблему, предложить решение, и вывести на удаление или дополнение отзыва в обмен на компенсацию.

### 1.2 Воронка общения

```
Покупатель оставил негативный отзыв (1-3★)
  → Мы открываем чат и пишем первое сообщение (триггер-фраза)
    → Узнаём, что не так (фаза "знакомство")
      → Обсуждаем проблему, даём обратную связь (фаза "понимание")
        → Предлагаем сделку: удаление отзыва = компенсация (фаза "решение")
          → Клиент удаляет → мы подтверждаем → выплачиваем компенсацию
```

### 1.3 Когда идём в чат

- Отзыв 1-3★ (4★ планируется, пока не активно)
- Товар активен (`work_status != 'not_working'`)
- Для товара включена работа в чатах (`product_rules.work_in_chats = true`)
- Для данной звёздности включен чат (`chat_rating_N = true`)
- Чат доступен на WB (`chat_status_by_review = 'available'` или `'opened'`)

### 1.4 Когда НЕ идём в чат

- Товар неактивен или не настроен для чатов
- Чаты отключены в WB-кабинете продавца (`chat_status_by_review = 'unavailable'`)
- Отзыв 4★ или 5★ (пока)
- Расширение ещё не открыло чат (нет `review_chat_link`)

### 1.5 KPI коммуникации

| Метрика | Цель |
|---------|------|
| CR диалог → удаление | Максимизировать |
| Средняя компенсация | Минимизировать (в рамках правил артикула) |
| Срок закрытия | Минимизировать |
| Конверсия ответов | Максимизировать |
| Доля эскалаций | Минимизировать |

## 2. Источники правил

### 2.1 Иерархия источников

| Приоритет | Источник | Что определяет | Где хранится |
|-----------|----------|----------------|-------------|
| 1 (высший) | Правила товара | Звёзды, стратегия, сумма компенсации | `product_rules` (БД) |
| 2 | Инструкции магазина | Тон, ограничения, FAQ | `stores.ai_instructions` (БД) |
| 3 | Настройки пользователя | Промпты AI, шаблоны цепочек | `user_settings` (БД) |
| 4 (базовый) | Hardcoded defaults | Fallback промпты и шаблоны | `src/ai/prompts/`, `src/lib/auto-sequence-templates.ts` |

### 2.2 Поля правил товара (`product_rules`)

```
Файл-источник: docs/database-schema.md
Код: src/db/helpers.ts (getProductRule, upsertProductRule)
```

| Поле | Тип | Описание |
|------|-----|----------|
| `work_in_chats` | boolean | Включить работу в чатах |
| `chat_rating_1` | boolean | Работаем с 1★ |
| `chat_rating_2` | boolean | Работаем с 2★ |
| `chat_rating_3` | boolean | Работаем с 3★ |
| `chat_rating_4` | boolean | Работаем с 4★ (планируется) |
| `chat_strategy` | enum | `upgrade_to_5` / `delete` / `both` |
| `offer_compensation` | boolean | Предлагать компенсацию |
| `max_compensation` | text | Фиксированная сумма в рублях (напр. "500") |
| `compensation_type` | text | `cashback` / `refund` |
| `compensation_by` | text | `r5` (по умолч.) / `seller` |

**Правило:** Сумма из `max_compensation` — это точная сумма предложения. Одинаковая для 1★, 2★ и 3★ по данному артикулу.

### 2.3 Инструкции магазина

```
Поле: stores.ai_instructions (TEXT)
API: GET/PUT /api/stores/{storeId}/ai-instructions
```

Свободный текст, инжектируется во все AI-потоки:
- Генерация ответов в чате
- Генерация предложения компенсации
- Ответы на отзывы
- Классификация чатов

Дополнительно загружаются:
- `store_faq` — Q&A для клиентов
- `store_guides` — инструкции (удаление отзыва, возврат)

### 2.4 Настройки авто-цепочек

```

```

| Поле | Назначение |
|------|-----------|
| `no_reply_messages` | Кастомные шаблоны (набор 1, негативы) |

## 3. Логика отбора отзывов

### 3.1 Воронка отбора

```
Отзыв (1-3★, в будущем 4★)
  │
  ├─ Товар активен? (work_status != 'not_working')
  │   └─ НЕТ → Пропустить
  │
  ├─ product_rules.work_in_chats = true?
  │   └─ НЕТ → Пропустить
  │
  ├─ chat_rating_N = true для этой звёздности?
  │   └─ НЕТ → Пропустить
  │
  ├─ chat_status_by_review?
  │   ├─ 'unavailable' → Чат недоступен (WB-кабинет не подключен)
  │   ├─ 'unknown' → Статус ещё не определён (расширение не парсило)
  │   ├─ 'available' → Чат доступен, можно открывать
  │   └─ 'opened' → Чат уже открыт
  │
  ├─ Есть review_chat_link?
  │   └─ НЕТ → Расширение ещё не открыло чат (задача для расширения)
  │
  ├─ Чат reconciled (chat_id заполнен)?
  │   └─ НЕТ → Ожидание следующего dialogue sync
  │
  ├─ Auto-launch проверки пройдены? (rating≤4, не resolved, нет buyer msgs, нет дублей)
  │   └─ НЕТ → Ожидание ручного запуска или следующего sync
  │
  └─ Авто-цепочка создана → Работа идёт (15 сообщений / ~30 дней для 1-3★, 10 / ~30 дней для 4★)
```

### 3.2 SQL-фильтр (TG-очередь)

Очередь в TG mini app показывает **только** связанные чаты:

```sql
SELECT c.*
FROM chats c
INNER JOIN review_chat_links rcl ON rcl.chat_id = c.id AND rcl.store_id = c.store_id
JOIN products p ON p.store_id = c.store_id AND c.product_nm_id = p.wb_product_id
JOIN product_rules pr ON p.id = pr.product_id AND pr.work_in_chats = TRUE
WHERE c.marketplace = 'wb' AND s.status = 'active'
```

```
Файл: src/db/telegram-helpers.ts
```

Это даёт ~700 чатов вместо ~300K+ (90% шумоподавление).

## 4. Структура ID заявки

### 4.1 Формат

**ID заявки на компенсацию = UUID-часть chat_id** (без префикса `1:`).

| Маркетплейс | Формат chat_id | Формат ID заявки |
|-------------|---------------|-----------------|
| WB | `1:{uuid}` | `{uuid}` |
| OZON | `{storeId}_ozon_{chatId}` | TBD |

UUID генерируется WB при открытии чата. Не создаётся нами.

### 4.2 Верификация

Скрипт `scripts/compensation-id-verify.mjs` проверил 423 сообщения с UUID:
- **422 совпадения** (99.8%) — UUID в тексте == UUID в chat_id
- **1 несовпадение** — ошибка копирования (ИП Авакова)

### 4.3 Связи данных

```
review_chat_links.review_id → reviews.id (отзыв)
review_chat_links.chat_id → chats.id (диалог)
chats.product_nm_id → products.wb_product_id (артикул)
chats.store_id → stores.id (кабинет / магазин)
chat_messages.chat_id → chats.id (сообщения)
```

### 4.4 Шаблон в сообщении

```
Укажите в сообщении ваш номер заявки на компенсацию: {chat_id_uuid}
```

> **Баг:** Сейчас в шаблоне есть суффикс `&tab=unread` — это артефакт URL WB-кабинета. Нужно убрать.

## 5. Стандарты коммуникации

### 5.1 Тональность

AI выбирает тон автоматически на основе контекста:

| Тон | Когда использовать | Примеры фраз |
|-----|-------------------|-------------|
| **empathetic** | Клиент расстроен, зол | "Понимаем, как это обидно...", "Нам правда жаль..." |
| **apologetic** | Явная ошибка продавца | "Это наша ошибка...", "Мы виноваты..." |
| **professional** | Нейтральная ситуация | "Спасибо за обращение...", "Готовы помочь..." |
| **friendly** | Клиент спокоен | "Спасибо, что написали!", "Рады, что вышли на связь!" |

**Общие правила:**
- Обращение на "вы"
- Использовать имя клиента (если известно)
- 2-4 предложения, без воды
- Тёплый живой тон, НЕ шаблонный

```
Файл: src/ai/prompts/deletion-offer-prompt.ts
```

### 5.2 Правила по фазам диалога

| Фаза | Ответов клиента | Что можно | Что нельзя |
|------|----------------|-----------|------------|
| **Знакомство** | 0 | Спрашивать что случилось, проявлять интерес | Предлагать компенсацию, упоминать удаление |
| **Понимание** | 1-2 | Реагировать на проблему, уточнять, давать обратную связь, задавать доп. вопросы | Называть конкретную сумму |
| **Решение** | 3+ | Предложить сделку: "с вас удаление — с нас кешбэк N₽", назвать сумму | Давить, шантажировать, обещать сверх лимита |

### 5.3 Рабочая стратегия (подтверждена владельцем)

Пошаговый алгоритм ведения диалога:

1. **Первый контакт:** Пишем, спрашиваем что не так. Проявляем интерес. Никаких предложений.
2. **Клиент ответил:** Уточняем детали, даём обратную связь по проблеме. Задаём ещё вопрос.
3. **Клиент написал подробнее:** Реагируем на конкретику. Говорим: "постараемся согласовать для вас бонус, дайте нам пару минут".
4. **Через некоторое время:** "Согласовали для вас вот такой вариант: с вашей стороны — удаление отзыва, с нашей — кешбэк {сумма}₽."

### 5.4 Абсолютные запреты

1. Давить на клиента или шантажировать
2. Обещать то, что не можем выполнить
3. Обвинять клиента в проблеме
4. Предлагать сумму выше указанной в правилах артикула (`max_compensation`)
5. Канцелярит и корпоративный жаргон
6. Игнорировать суть проблемы клиента
7. Предлагать компенсацию, если клиент ещё не ответил ни разу

### 5.5 Стратегии по товарам

Настраивается в `product_rules.chat_strategy`:

| Стратегия | Описание | Пример фразы (фаза "решение") |
|-----------|----------|------------------------------|
| `upgrade_to_5` | Мягкий намёк на пересмотр оценки до 5★ | "Если после этого вы почувствуете, что ситуация решена — будем рады, если пересмотрите оценку" |
| `delete` | Прямое предложение удаления за компенсацию | "Согласовали: с вас удаление отзыва, с нас кешбэк {сумма}₽" |
| `both` | Универсальный подход — фокус на решении проблемы | "Хотим, чтобы негативный опыт не остался последним впечатлением" |

## 6. Шаблоны сообщений

> **Важно:** Шаблоны WB и OZON — раздельные. Ниже описаны шаблоны WB. Шаблоны OZON в разработке.

### 6.1 Этап 1 — Первичный контакт (триггер-фраза)

**Когда:** Расширение открывает чат по отзыву 1-3★ и отправляет первое сообщение.

**WB (default):**
> Здравствуйте! Мы увидели ваш отзыв и хотели бы узнать подробнее — расскажите, пожалуйста, что именно вас расстроило?

**Переменные:** нет (статический текст)
**Override:** `user_settings.no_reply_trigger_phrase`

```
Файл: src/lib/auto-sequence-templates.ts → DEFAULT_TRIGGER_PHRASE
```

**Что происходит после отправки:**
- Чат получает тег `deletion_candidate`
- Статус → `in_progress`
- Создаётся `chat_auto_sequence` с 14 шаблонами
- Cron каждые 30 мин проверяет и отправляет следующее сообщение

### 6.3 Этап 3 — Предложение компенсации (AI-генерация)

**Когда:** Клиент ответил 3+ раз, диалог достиг фазы "решение".

**AI генерирует персонализированное сообщение** на основе:
- Имя клиента, название товара
- История диалога (что клиент рассказал)
- Сумма компенсации из `product_rules.max_compensation`
- Стратегия из `product_rules.chat_strategy`
- Инструкции магазина из `stores.ai_instructions`

**Пример (стратегия `delete`):**
> Спасибо за терпение и за то, что рассказали о ситуации. Мы согласовали для вас кешбэк {сумма}₽. С вашей стороны — удаление отзыва. Как вам такой вариант?

**Пример (стратегия `upgrade_to_5`):**
> Спасибо, что рассказали подробнее. Нам правда жаль, что так получилось. Готовы вернуть {сумма}₽ на карту. Если после этого почувствуете, что ситуация решена — будем рады, если пересмотрите оценку.

```
Файл: src/ai/flows/generate-deletion-offer-flow.ts
Промпт: src/ai/prompts/deletion-offer-prompt.ts
```

### 6.4 Этап 4 — Подтверждение удаления и компенсация

**Процесс (пока частично ручной):**

1. Клиент пишет в чат: "Отзыв удалён" (или аналогичное)
2. Система делает API запрос к WB для проверки `review_status_wb`
3. Менеджер проверяет вручную (визуально подтверждает)
4. В TG mini app менеджер нажимает "записать в отчёт" → выбирает "удалён" или "дополнен"
5. Формируется сообщение покупателю:

**Шаблон подтверждения (WB):**

> Видим, что отзыв был удалён. Благодарим вас! Напишите, пожалуйста, нашему руководителю клиентского сервиса в Макс или Телеграмм по номеру +79950509459, мы оформим компенсацию в удобном для вас формате.
>
> Укажите в сообщении ваш номер заявки на компенсацию: {chat_id_uuid}

**Переменные:**
- `{chat_id_uuid}` — UUID-часть chat_id (без префикса `1:`)

**Контакт R5:** +79950509459 (актуальный). В будущем — настраиваемый в ПО.

### 6.5 Этап 5 — Мягкий выход (стоп-сообщение)

**Когда:** 14 дней прошло, клиент не ответил ни разу.

**WB, негативы (1-3★):**
> Здравствуйте! Мы так и не получили от вас ответа, но хотим, чтобы вы знали — ваш отзыв не остался без внимания.
>
> Мы уже учли обратную связь по этому товару и работаем над улучшениями. Если в будущем захотите вернуться к разговору — мы всегда на связи.
>
> Спасибо, что нашли время поделиться впечатлениями. Хорошего дня!

**WB, 4★ (планируется):**
> Здравствуйте! Видим, что вы не смогли ответить — ничего страшного.
>

## 7. Политика компенсаций

### 7.1 Принцип

**Компенсация фиксированная по артикулу.** Одна и та же сумма для 1★, 2★ и 3★ — строго по правилам товара.

### 7.2 Параметры

| Параметр | Значение |
|----------|----------|
| Источник суммы | `product_rules.max_compensation` |
| Принцип расчёта | Фиксированная сумма по артикулу, НЕ зависит от рейтинга |
| Практический диапазон | от ~100₽ до ~2000₽ |
| Жёсткие лимиты | Нет. Строго по правилам артикула. |

### 7.3 Кто платит

| Значение `compensation_by` | Описание | Пример |
|---------------------------|----------|--------|
| `r5` (по умолчанию) | R5 выплачивает компенсацию клиенту | Большинство магазинов |
| `seller` | Продавец платит сам | Пацюк (планируется расширение) |

### 7.4 Типы компенсации

| Тип | Описание | Как оформляется |
|-----|----------|----------------|
| `cashback` | Кешбэк (перевод на карту) | Клиент пишет в WhatsApp/Telegram по номеру R5 |
| `refund` | Полный возврат стоимости товара | Оформляется через WB |

### 7.5 Фазы предложения

| Фаза | Ответов клиента | Действие менеджера |
|------|----------------|-------------------|
| Знакомство | 0 | **НЕ предлагать** компенсацию. Только вопросы. |
| Понимание | 1-2 | Намекнуть: "постараемся согласовать бонус". Без суммы. |
| Решение | 3+ | Конкретная сделка: "удаление отзыва = {сумма}₽ кешбэк" |

### 7.6 Эскалация

- Клиент требует сумму выше `max_compensation` → передать руководителю
- Клиент хочет другой тип компенсации → уточнить у руководителя

### 7.7 Отказ

- Клиент агрессивен, угрожает → тег `unsuccessful`, прекратить предложения
- Спам / конкуренты → тег `spam`, игнорировать
- Клиент не заинтересован → мягкий выход (стоп-сообщение)

## 8. Алгоритм рассылки

### 8.1 Текущий режим

| Параметр | Значение |
|----------|----------|
| Длительность | 14 дней |
| Частота | 1 сообщение в день |
| Сообщений всего | 14 |
| Время отправки (MSK) | 10:00-17:00, weighted slots |
| Cron-интервал | Каждые 30 мин (8:00-22:00 MSK) |
| Макс сообщений/день/чат | 1 |
| Rate limit | 3 сек между отправками |

### 8.2 Планируемый режим (новая рассылка)

| Параметр | Значение |
|----------|----------|
| Длительность | 30 дней |
| Частота | Раз в 3 дня |
| Сообщений всего | ~10 |
| Остальное | Аналогично текущему |

### 8.3 Распределение по времени (MSK)

```
10:00 — 15%    14:00 — 10%
11:00 — 15%    15:00 — 10%
12:00 — 15%    16:00 — 10%
13:00 — 15%    17:00 — 10%
```

Внутри часа — случайная минута (0-59) для scatter.

```
Файл: src/lib/auto-sequence-templates.ts → SEND_SLOTS
```

### 8.4 Алгоритм обработки (cron)

```
ДЛЯ КАЖДОЙ pending последовательности:
│
├─ 1. Клиент ответил после start? → stopSequence('client_replied')
│
├─ 2. Статус чата валидный (inbox/in_progress/awaiting_reply)?
│   └─ НЕТ → stopSequence('status_changed')
│
├─ 3. Seller уже писал сегодня?
│   └─ ДА → reschedule на завтра (случайный слот)
│
├─ 4. Достигнут max_steps (14)?
│   ├─ ДА → отправить стоп-сообщение
│   ├─ Статус чата → 'closed', completion_reason → 'no_reply'
│   └─ completeSequence()
│
├─ 5. Отправить шаблон[current_step]
│   ├─ Записать в chat_messages (is_auto_reply=true)
│   ├─ Обновить last_message_* на чате
│   └─ advanceSequence()
│
└─ 6. Пауза 3 сек → следующая последовательность
```

```
Файл: src/lib/cron-jobs.ts → startAutoSequenceProcessor()
```

### 8.5 Стоп-условия

| Условие | Действие | Поле reason |
|---------|----------|------------|
| Клиент ответил | Остановить цепочку | `client_replied` |
| Статус чата изменён (не inbox/in_progress/awaiting) | Остановить | `status_changed` |
| 14 дней прошло (max_steps) | Стоп-сообщение → close | `no_reply` |

### 8.6 Дедупликация

- Проверка `chat_messages` за сегодня: если seller уже писал → пропуск, reschedule на завтра
- Одна авто-цепочка на чат (не создаётся повторно)

## 9. Работа с рисками

### 9.1 Таблица рисков

| Риск | Признак | Действие |
|------|---------|----------|
| **Агрессивный покупатель** | Оскорбления, угрозы, ультиматумы | Тег `unsuccessful`, прекратить предложения, эскалация |
| **Подозрение на скриншот** | Клиент может заскриншотить и пожаловаться | В начале — НЕ упоминать удаление. Только после установки контакта и доверия. |
| **Провокация / конкурент** | ALL CAPS, внешние ссылки, подозрительный текст | Тег `spam` (ALL CAPS = 100% спам), игнорировать |
| **Угроза жалобы / суд** | "Напишу жалобу", "роспотребнадзор", "суд" | Профессиональный тон, не давить, эскалация руководителю |
| **Клиент требует больше** | Сумма выше `max_compensation` | Эскалация на руководителя, НЕ обещать сверх лимита |
| **Двойная отправка** | Баг или race condition | Дедупликация: seller sent today → skip |
| **Клиент просит возврат** | "Верните деньги" без упоминания отзыва | Направить на стандартный возврат WB *(тег `refund_requested` удалён, migration 024)* |

### 9.2 Классификация рисков (AI + Regex)

```
Файл: src/ai/flows/classify-chat-deletion-flow.ts
Триггеры: docs/domains/DELETION_TRIGGER_PHRASES.md
```

**Двухстадийная классификация:**

1. **Stage 1: Regex (быстро)** — проверка ключевых фраз
   - "удалю отзыв" → 98% confidence → `deletion_candidate`
   - "поставлю 5 звезд" → 96% confidence → `deletion_candidate`
   - ~~"верните деньги" → 92% confidence → `refund_requested`~~ *(удалён, migration 024)*
   - ALL CAPS → 100% → `spam`

2. **Stage 2: AI (если regex < 90%)** — глубокий анализ контекста
   - Использует Deepseek Chat
   - Учитывает историю диалога, тон, намерение

## 10. Метрики эффективности

### 10.1 Основные метрики

| Метрика | Описание | Как считать |
|---------|----------|-------------|
| **CR диалог → удаление** | % связанных чатов, где отзыв удалён | `COUNT(deletion_confirmed) / COUNT(linked_chats)` |
| **Средняя компенсация** | Средний кешбэк по закрытым чатам | `AVG(max_compensation) WHERE tag='deletion_confirmed'` |
| **Срок закрытия** | Дни от первого сообщения до подтверждения удаления | `confirmed_at - first_message_at` |
| **Конверсия ответов** | % клиентов, которые ответили на рассылку | `COUNT(client_replied) / COUNT(total_sequences)` |
| **Доля эскалаций** | % чатов с тегом unsuccessful | `COUNT(unsuccessful) / COUNT(linked_chats)` |
| **Отказы** | % чатов → стоп без ответа | `COUNT(no_reply_completed) / COUNT(total_sequences)` |

## 12. Глоссарий

| Термин | Определение |
|--------|------------|
| **Чат** | Диалог с покупателем на маркетплейсе (WB / OZON). Открывается через расширение. |
| **Компенсация** | Выплата покупателю за удаление/дополнение негативного отзыва. Фиксированная по артикулу. |
| **Кешбэк** | Тип компенсации — перевод на карту клиента. |
| **Возврат (refund)** | Тип компенсации — полный возврат стоимости товара через WB. |
| **Кабинет** | Магазин продавца на WB/OZON. Один продавец может иметь несколько кабинетов. |
| **Активный товар** | Товар с `work_status != 'not_working'` и настроенными `product_rules`. |
| **Триггер-фраза** | Первое сообщение продавца, которое запускает авто-цепочку и тег `deletion_candidate`. |
| **Авто-цепочка** | Серия автоматических сообщений (14 шт. за 14 дней), отправляемых если клиент не отвечает. |
| **Стоп-сообщение** | Финальное сообщение после 14 дней без ответа. Закрывает чат. |
| **review_chat_links** | Таблица связей между отзывами и чатами. Заполняется расширением. |
| **Reconciliation** | Процесс привязки `chat_id` из WB API к записи `review_chat_links`. Происходит при dialogue sync. |
| **deletion_candidate** | Тег чата: AI обнаружил возможность удаления/изменения отзыва. |
| **chat_strategy** | Стратегия работы по товару: `upgrade_to_5` (повысить до 5★), `delete` (удалить), `both` (универсально). |
| **Эскалация** | Передача чата руководителю при превышении лимитов, агрессии клиента, или нестандартной ситуации. |
| **ID заявки** | UUID-часть chat_id. Используется для идентификации при оформлении компенсации. |
| **Фаза диалога** | Этап общения: знакомство (0 ответов) → понимание (1-2) → решение (3+). |
| **max_compensation** | Поле в `product_rules`. Фиксированная сумма компенсации для артикула. |
| **Dialogue sync** | Cron-процесс синхронизации чатов с WB API. Каждые 5-60 мин. |
| **work_in_chats** | Флаг в `product_rules`. Включает/выключает работу в чатах для товара. |
| **TG mini app** | Telegram мини-приложение для менеджеров. Очередь чатов, действия, уведомления. |

## ЗОНЫ ДЛЯ АВТОМАТИЗАЦИИ

### 1. Авто-подтверждение удаления

**Сейчас:** Менеджер вручную проверяет удаление и формирует сообщение.
**Предложение:** Клиент пишет "удалил" → cron проверяет `review_status_wb` через API → если `deleted` → авто-формирование сообщения с ID заявки → менеджер подтверждает одной кнопкой в TG.
**Эффект:** -80% ручной работы на этапе подтверждения.

### 2. Настраиваемый контакт R5

**Сейчас:** Номер +79950509459 захардкожен в ручных шаблонах.
**Предложение:** Вынести номер в `stores` или `user_settings`. При смене — обновление во всех шаблонах автоматически.
**Эффект:** Готовность к масштабированию (разные номера для разных регионов/команд).

### 3. Авто-эскалация по ключевым словам

**Сейчас:** AI классифицирует, но не оповещает о критических ситуациях.
**Предложение:** При обнаружении "жалоба", "суд", "роспотребнадзор", "Wildberries модерация" → тег `escalation` + мгновенное TG-уведомление владельцу.
**Эффект:** Раннее обнаружение рисков, предотвращение репутационных потерь.
